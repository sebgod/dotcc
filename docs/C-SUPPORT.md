# C language support in dotcc

Running tracker of what dotcc's grammar and libc cover today. Update this when a feature lands (add the fixture / test reference in **Notes**), when one moves from вќЊ to рџџЎ to вњ…, or when something is decided out of scope (рџљ« with reason).

**Default dialect: `c17`.** Select otherwise with `-std=<dialect>` (`c90`/`c99`/`c11`/`c17`/`c18`/`c23`) вЂ” see the CLI flag table in CLAUDE.md. `-std=` predefines `__STDC_VERSION__` (and friends) so headers can branch via `#if __STDC_VERSION__ >= вЂ¦L`, and drives rule-2 keyword promotion (`inline` becomes a keyword в‰Ґc99; `bool`/`true`/`false`/`nullptr`/`noreturn`/`typeof` only в‰Ґc23 вЂ” gated on the monotonic `CDialect.Version`, which is keyed by ISO year). **By itself `-std=` stays permissive** вЂ” the parser is dialect-agnostic, so `//` comments, `_Bool`, designated initializers etc. are accepted regardless. **`-pedantic` / `-pedantic-errors` opt into the *rejection* gate** (gcc model): features newer than the selected `-std=` are diagnosed as warnings / errors. So the year tags below (`C99` / `C11` / `C23`) are descriptive for the default path, and become enforcement gates under `-pedantic`. Gated today: `_Bool` / `long long` / `ll`-suffix / designated init / `for`-init decl / `__func__` / variadic macros / mixed declarations-and-statements / compound literals / hex float literals / array designators / `_Complex` / flexible array member (C99), `_Static_assert` / `_Noreturn` / anonymous struct/union members (C11), `enum : T` / `_Float128` / `#warning` / empty initializer `{}` / message-less `_Static_assert(expr)` (C23). Not gated: `//` comments (universal extension); K&R definitions are moot (not parsed) and VLAs are out of scope. Always-modern *output* is unaffected by `-std=` вЂ” dotcc emits real C# enums etc. regardless of input dialect.

**No known silent miscompiles** вЂ” every gap fails loudly at parse/lex time, and `-pedantic` turns dialect violations into diagnostics.

## Lexical

| Feature | Status | Notes |
|---|---|---|
| Line comments `// вЂ¦` | вњ… | `LINE_COMMENT` lexer rule, action `ignore` |
| Block comments `/* вЂ¦ */` | вњ… | `BLOCK_COMMENT` lexer rule, the canonical C-comment regex `/\*([^*]\|\*+[^*/])*\*+/` (alternation, supported since LALR.CC в‰Ґ 4.0.0). A run of stars before the slash belongs to the terminator, so a comment closed with `**/` (Lua's doc-comment style, e.g. lfunc.c's `newupval`) ends there. (The earlier no-alternation form `(\*[^/][^*]*)*` mis-scanned `**/` вЂ” the slash fell through into the body вЂ” and ran the comment on to the next `*/`, swallowing the code between.) Fixture `block-comment-close/`, unit tests `BlockCommentTests`. |
| Identifiers `[a-zA-Z_][a-zA-Z_0-9]*` | вњ… | `ID` token. A C identifier that collides with a **C# reserved keyword** (`new`, `lock`, `is`, `string`, `this`, `ref`, `object`, `in`, `out`, вЂ¦) is `@`-escaped on emit (`@new`) вЂ” consistently at every declaration and reference (locals, params, function names + calls, struct fields + member access, labels, enum constants), since the escape is a pure function of the name. Fixtures `cs-keyword-idents/`, `keyword-true-false-ident/`. **Not escaped:** `null` (dotcc emits it as the bare C# `null` literal вЂ” the only expression that implicitly converts to any pointer type вЂ” so a variable named `null` is a residual edge; `true`/`false` are NO LONGER in this bucket вЂ” they lower to `1`/`0` and so escape correctly), and **type names** (struct/enum/union tags, typedef-names) вЂ” a `struct lock` tag isn't escaped yet. |
| Decimal int literal | вњ… | `NUM` token |
| Hex int literal `0xFF` | вњ… | Lexer rule above NUM (longest-match wins); visitor passes through вЂ” C# accepts identical syntax. Fixture `bitwise/` |
| Octal int literal `0755` | вњ… | Lexed by the decimal `NUM` rule (a `0`-prefixed run), then **converted** in `Visit(C.Num)` to its value (`0755` в†’ `493`) вЂ” C# has no octal syntax (a leading `0` is plain decimal there), so emitting verbatim would silently mean 755. Digits validated `0`вЂ“`7` (`0789` в†’ clear error). C89, no dialect gate. Fixture `octal-binary-literals/`. |
| Binary int literal `0b1010` | вњ… | C23. Lexer rules `0[bB][01]+` (В± suffix) above the decimal rule; C# accepts `0bвЂ¦` verbatim so the visitor passes it through. Gated as C23 under `-pedantic`. Fixture `octal-binary-literals/`. |
| Digit separators `1'000'000` | вњ… | C23. Each NUM digit-run rule allows a single `'` between digits (`D('?D)*` вЂ” never leading/trailing/doubled), in decimal, hex, and binary. The visitor strips them (C# uses `_` and needs none) before evaluating. Gated as C23 under `-pedantic`. Fixture `digit-separators/`. |
| Int suffixes `123u`, `123L`, `123ull` | вњ… | Lexer accepts any combination of `u`/`U`/`l`/`L` (one or more) after a decimal/hex/binary/octal digit run. Visitor's `CsIntSuffix` collapses to C#'s form: `u` в†’ `u` (compiler resolves uint/ulong), one-or-more `L` в†’ `L` (long; C# has no `ll`), `u` + `L`s в†’ `UL` (ulong). |
| Float literal `1.5`, `1.5e10`, `1.5f` | вњ… | `FLOAT` token (mandatory `.` to disambiguate from `NUM`) |
| Float literal suffix `1.5L` (long double) | вњ… | The suffixed FLOAT lexer rule accepts `[fFlL]`. `f`/`F` passes through to C# verbatim (float); `l`/`L` is C's long-double suffix в†’ `Visit(C.Flt)` strips it and emits a bare `double` literal (mirroring dotcc's `long double` в†’ C# `double` mapping вЂ” C# has no float `L` suffix). Works after an exponent too (`6.0e2L`). An `f`-suffixed constant is **typed `float` in the IR** (В§6.4.4.2), not double-with-a-suffix вЂ” so `double x = 1.5f;` is a float value promoted at the store (the wat backend renders `f32.const` + `f64.promote_f32`, exactly clang's lowering), and a `float f = 1.5f;` init needs no cast. Fixture `float-literal-suffix/` (gcc-oracle-validated). |
| Hex float literal `0x1.8p3` (C99) | вњ… | `0xH.HpE` вЂ” value = (hex mantissa) Г— 2^(decimal exponent); the `p`/`P` exponent is mandatory (so longest-match cleanly separates it from a hex int). Three FLOAT lexer rules cover the int-part/fraction-part variations. C# has **no** hex-float literal, so `Visit(C.Flt)` parses the value (`ParseHexFloat`) and emits a decimal `double` (or `float` for the `f` suffix; `l`/`L` в†’ double). Exactly-representable constants round-trip bit-for-bit; extreme exponents could differ by an ULP via `Math.Pow`. Gated C99 under `-pedantic`. Fixture `hex-float-literals/` (gcc-oracle-validated). |
| Char literal `'a'`, `'\n'`, `'\033'`, `'\x1b'` | вњ… | Five `CHAR` lexer rules (longest-match: hex `'\xNN'`, 3- and 2-digit octal, single-escape incl. 1-digit octal, plain). The visitor **decodes** the escape to its byte value: a plain printable char stays `(byte)'c'`, everything else (named escape, octal, hex, control) в†’ `(byte)N`. Full escape set: `\a \b \e \f \n \r \t \v \0 \NNN \xNN \\ \' \" \?` (`\e` is the GNU ESC). Char bytes 0вЂ“255 all fine (it's a `(byte)` cast). Fixtures `small-ops/`, `string-char-escapes/`. |
| String literal `"вЂ¦"` + adjacent concatenation `"a" "b"` | вњ… | `STRING` token regex `"(\\["\\nrtbf0v'/aex0-9]\|[^"\\])*"` (open quote, then backslash-escapes or non-quote-non-backslash, then close). **Adjacent literals concatenate** (C phase 6) via a left-recursive `StringSeq` в†’ one `Primary`. Each segment's escapes are **decoded independently** to bytes, then concatenated, then re-emitted as a single greedy-safe C# `u8` literal: named escapes (`\n \t \r \\ \"`), and everything else as `\xHH` вЂ” and a `\xHH` is never left adjacent to a literal hex digit (C#'s `\x` is greedy), so octal/hex escapes and `"\x1b" "abc"`-style boundaries are correct. Non-ASCII source chars pass through (the u8 literal UTF-8-encodes them, matching C's UTF-8 source bytes). Fixtures `string-escape-quotes/`, `string-char-escapes/`. **High bytes:** a decoded escape byte > 0x7F (`"\xff"`, `"\377"`) can't be a single byte in a u8 literal (C# UTF-8-encodes it to two), so such strings lower to a constant byte-array `L(new byte[]{ 0xFF, вЂ¦, 0 })` carrying the exact C bytes вЂ” Roslyn RVA-optimizes a constant `byte[]` in `ReadOnlySpan<byte>` position, so `L()` still pins a fixed-address pointer (no alloc, no GC move), identical to the u8 case. All-ASCII strings keep the readable u8 literal. Fixture `string-high-byte-escapes/` (gcc-oracle-validated). Relies on IRx alternation in LALR.CC. |
| `char16_t` + `u"вЂ¦"` / `u'x'` literals (C11) | вњ… | `char16_t` (`<uchar.h>`) в†’ C# `char` вЂ” a 16-bit UTF-16 code unit вЂ” so `char16_t*` walks 2 bytes and the type bridges to C# `string`/`Span<char>`. `u"вЂ¦"` в†’ a pooled, pinned `Libc.L16` `char*` (UTF-16, NUL-terminated); `u'x'` в†’ a `char16_t`-typed value; `char16_t s[] = u"вЂ¦"` copies to a `char` stackalloc (local) / `Libc.GlobalArrayFrom<char>` (global) / zero-copy RVA `Libc.L<char>` (const). | `wchar_t` + `L"вЂ¦"` / `L'x'` literals | вњ… | `wchar_t` (`<wchar.h>`) в†’ C# `char` вЂ” dotcc's **MSVC-shaped 16-bit** wchar_t (a deliberate, documented ABI choice; `__SIZEOF_WCHAR_T__ == 2`). ## libc (`DotCC.Libc/`)

The runtime surface dotcc-emitted programs link against. Each function routes to a BCL primitive. Implementations live in `DotCC.Libc/*.cs`; the same source compiles into `DotCC.Libc.dll` for unit tests AND is embedded as a resource into `DotCC.Lib.dll` (spliced into every emitted program by `BuildShell` вЂ” single source of truth). Once `DotCC.Libc` is published to NuGet, the embedding goes away and emitted programs reference the package directly.

**Design rule: reentrant by default.** Where the C standard offers a stateful and a reentrant variant of the same function (`strtok` / `strtok_r`, `rand` / `rand_r`, `asctime` / `asctime_r`, `localtime` / `localtime_r`, etc.), dotcc implements the reentrant form as the primitive and exposes the stateful form (if at all) as a thin wrapper using `[ThreadStatic]` storage. Hidden static state in libc is a notorious source of multithreading bugs; emitted dotcc programs target .NET where threading is cheap and idiomatic, so the reentrant shape is the right default. Real C programs that need the C89 stateful form keep working through the wrapper, but the docs steer users to the `_r` variant.

### `stdio.h`

**FILE model.** A C `FILE` is a tiny opaque struct (`Libc.FILE`, one slot index) and `FILE*` stays a **genuine pointer** вЂ” so `NULL`, `==`, and `if (fp)` work through the normal pointer machinery with no emitter special-casing (`FILE` is seeded in `Compiler.PredefinedTypeNames`, like `div_t`). The slot indexes a managed table (`FileSlot`): slots 0/1/2 are `stdin`/`stdout`/`stderr` and route text through `Console.In`/`Out`/`Error` **resolved per call** (so redirection / the functional harness's capture are honored вЂ” never the raw OS handle); `fopen`'d slots wrap a `FileStream` (binary, seekable). `freopen` swaps a slot's backing while keeping the same `FILE*` identity вЂ” the standard way (C99 7.19.5.4) to point `stdin`/`stdout` at a file. Implementations in `FileLib.cs`; unit tests `LibcFileTests.cs`; fixture `file-io-roundtrip/`.

Token identity disambiguates nested/frame-skipping captures here too. **Limitation**: setjmp buried in a loop/ternary condition, a nested sub-expression (`x = setjmp(env) + 1`), or a bare discarded call (`setjmp(env);`) is **rejected loudly** with a `CompileException` naming the supported shapes вЂ” never silently lowered to the always-0 runtime stub (which would turn a `longjmp` into an uncaught exception). A value-capture whose target is a non-trivial lvalue (`s.field = setjmp(env)`) is likewise rejected (V1: VarRef targets only). `jmp_buf` lowers to the predefined C# type `Libc.LongJmpToken` via the TypeNameRewriter seed list вЂ” no fake keywords in the grammar. Fixtures `setjmp-cleanup/` (3-deep frame unwind, with-else), `setjmp-no-else/` (the `LUAI_TRY` no-else shape), `setjmp-nested/` (inner `longjmp` to the outer env, gcc-oracle-validated), `setjmp-switch-value/` (bare `switch (setjmp(env))` dispatching distinct codes), and `setjmp-capture-if/` (`int r = setjmp(env)` + `if` on the value); unit tests `SetjmpValueCaptureTests` pin each shape and each rejection. |

### POSIX headers (`unistd.h`, `fcntl.h`, `poll.h`, `signal.h`, `sys/*`)

The minimal POSIX surface portable C programs probe for (chibi-scheme's non-Plan9/non-Windows path compiles against exactly this set). Implementations are split by area: `DotCC.Libc/PosixFsLib.cs` (filesystem + `dirent` + `open`), `PosixLib.cs` (`sys/stat`/`sys/time`/`fcntl`/`poll`/`socket`/`getrusage`), `ProcessSignalLib.cs` (process control + signals), `UnistdLib.cs` (`usleep`/`isatty`/`select` surface), and the fd primitives in `FileLib.cs`. The `sys/` headers embed under their slash-qualified names (`DotCC.SystemHeaders.sys/stat.h`) so `#include <sys/stat.h>` resolves like any synthetic header.

**This whole surface is honest about its holes вЂ” there are no silent lies.** Every entry falls in one of four fidelity tiers, and the tier is the point of the table below: a program that strays into a hole gets a loud, conformant failure (в€’1 + `errno`, or a thrown exception), never a fake success.

| Tier | Meaning |
|---|---|
| вњ… **faithful** | Behaves as POSIX. Either a direct .NET equivalent (`System.IO`, `Environment`), or вЂ” where .NET has no managed API but the OS does вЂ” a P/Invoke to the **real OS primitive** behind an `OperatingSystem.IsWindows()` switch (`link`в†’`link(2)`/`CreateHardLinkW`, `getppid`в†’`getppid(2)`/toolhelp snapshot, `getrusage`в†’`getrusage(2)`/`GetProcessTimes`). Those P/Invokes are blittable (pointers + ints), so `[DllImport]` emits a direct AOT-clean call with no marshalling stub вЂ” and, unlike `[LibraryImport]`, needs no source generator, so it survives the runtime-block splice into the functional tests' generator-less Roslyn compile. |
| рџџЎ **best-effort** | Works for the common case; a documented divergence outside it (stated in **Notes**). Conformant-but-degraded, never wrong. |
| вљЄ **inert** | Accepted and ignored: returns success with no observable effect. Safe because dotcc's execution model makes the effect a no-op (single managed process, file-backed fds, runtime-owned signals). |
| вќЊ **no primitive** | No .NET equivalent exists, so the call **fails loudly** вЂ” `errno` + в€’1 (or `throw`), exactly what a real libc returns when the operation can't be done. The hole is visible at the call site, not hidden. |

## Remaining syntax gaps, by dialect

A consolidated quick-reference of C *syntax* (grammar) not yet supported, grouped
by the standard that introduced it. The detailed per-feature rows above are the
source of truth; this is the at-a-glance roadmap. **Severity:** в›” parse/lex
error (fails loudly вЂ” safe, never a wrong answer) В· рџџЎ partial В· вќЊ roadmap
(would-implement) В· рџљ« deliberately out of scope. **There are no known silent
miscompiles** вЂ” every gap below fails at parse/lex time or is explicitly flagged.

**C89 / C90** вЂ” **complete** (every parseable C89 syntax feature is supported).
- рџљ« K&R function definitions; `register` storage class. (`auto` вњ… вЂ” storage-class dropped, C23 inference в†’ `var`.) Nested-brace init handles the regular fully-flat and fully-nested shapes with C's zero-fill; irregular *mixed* brace/scalar elision raises a clear error rather than miscompile.

**C99**
- вњ… Compound literals вЂ” `(struct S){вЂ¦}` (struct/union, any position); `(int){5}` (scalar в†’ cast, any position); `(int[]){вЂ¦}` (array вЂ” scalar/struct-element/designated element forms; in initializer position в†’ stackalloc, elsewhere hoisted to a block-local pointer temp so it works as a call arg / return / subscript). See the Types table.
- вњ… Array designators `[2] = 5` вЂ” `int a[5] = {[2]=9, [4]=1}` (1-D scalar; dense zero-filled compile-time build, running cursor + reorder + implicit-size; struct/nested-designated still в›”). Gated C99; fixture `array-designators/`.
- вњ… Hex float literals `0x1.8p3` вЂ” parsed + converted to a decimal value (see the Lexical table).
- вњ… `_Complex` вЂ” `float`/`double`/`long double _Complex` в†’ `System.Numerics.Complex` (arithmetic via Complex's operators + implicit double conversion, the imaginary unit `I`, and the `<complex.h>` surface). `float`/`long double` widen to double (documented). - вњ… Flexible array members `struct { int n; int data[]; }` (C99) вЂ” and sized array members `int grid[N];` (C89, which also didn't parse before). Both lower to a C# `fixed`-size buffer. See the Types table.
- рџљ« VLAs вЂ” out of scope (rare, optional since C11, managed-runtime mismatch). A 1-D runtime extent incidentally lowers to `stackalloc`, but it's not a pursued feature.
- вњ… `inline` вЂ” function specifier в†’ `[MethodImpl(AggressiveInlining)]` (see the Declarations table).

**C11**
- вњ… `_Generic` (generic selection) вЂ” resolved at lowering time on the controlling expression's synthesized `CType`. See the Beyond-C99 table.
- вњ… Anonymous `struct` / `union` members вЂ” fields promoted into the parent (struct inlined; union lifted to a nested explicit-layout type + access rewrite). See the Beyond-C99 table.
- вњ… `_Thread_local` (в†’ `[ThreadStatic]`, zero-init file-scope). (`_Atomic` + full `<stdatomic.h>` вњ…, `_Noreturn` вњ…, `_Static_assert`/`static_assert` вњ… evaluated at compile time, `_Alignof` вњ… folds / `_Alignas` вњ… checked-then-ignored вЂ” see the Beyond-C99 table.)

## Dialect gating (rewriter / grammar / visitor)

How dotcc decides whether a feature is available under a given `-std=` вЂ” the
model folded in from the former `HANDOVER-std-option.md`. `-std=` predefines
`__STDC_VERSION__` and drives keyword promotion; by itself it stays
**permissive** (the parser is a superset of every dialect), and `-pedantic` /
`-pedantic-errors` turn the per-feature year tags into warnings / errors. Three
layers carry the gating; the right one depends on the feature's *shape*:

1. **Rewriter** (`DialectKeywordRewriter`) вЂ” a **new keyword spelled like an
   identifier** (`bool`, `true`, `false`, `nullptr`, `static_assert`, `noreturn`,
   `typeof`). These are *legal identifiers* in older dialects (`int true = 5;` is
   valid C99), so the grammar can't always treat them as keywords вЂ” it would
   parse-error on valid old code before the visitor runs. The `ID в†’ keyword`
   promotion itself carries the `MinVersion`: promoted at/above the introducing
   standard, left an identifier below it (which also lets the pre-C23
   `<stdbool.h>` / `<stddef.h>` macro supply the meaning). One `MinVersion` covers
   both directions at once. (`inline` is the same shape, gated в‰Ґc99.)
2. **Grammar** (`c.lalr.yaml`) вЂ” **genuinely new syntax** that can't collide with
   an identifier (`_BitInt(N)`, `typeof`, `_Generic`, `[[attributes]]`). Parsed
   always (the superset); *availability* gated in the visitor.
3. **Visitor** (`CSharpEmitter`, via `DialectGate`) вЂ” *restrictions*: a construct
   the grammar accepts but an older `-std=` shouldn't (designated init / `long
   long` under c90, вЂ¦) becomes a `-pedantic` diagnostic. Also semantic *flips*
   like C23 `auto` (storage class в†’ type inference), branching on the threaded
   `CDialect`.

`_Capital_` keywords (`_Bool`, `_Static_assert`, `_Noreturn`, `_Atomic`,
`_Alignas`, вЂ¦) are reserved to the implementation in *every* dialect, so they
**always lex as keywords**; their *availability* can still be visitor-gated. We
deliberately keep dialect awareness OUT of LALR.CC itself вЂ” a single client
doesn't justify the table-doubling; revisit only with a second
standards-versioned client.

**Residual keyword TODOs** (status mirrors the Beyond-C99 / dialect-gaps tables):
- `_Thread_local` / `thread_local` (C11) вЂ” **not built; weigh value first.** No
  faithful lowering in the current emit shape: `[ThreadStatic]` is field-only
  (no per-local TLS). It'd only stub. (`_Alignas` / `_Alignof` вњ… вЂ” the fold +
  checked-no-op landed with the comptime-dividend milestone; there's still no
  per-local alignment hook on a `stackalloc`, which is exactly why `_Alignas`
  is accept-and-ignore.)
- `_Generic` (C11) вЂ” the common case (`tgmath.h`) is already covered by C#
  overload resolution; low priority.
- `constexpr` (C23) needs a const-eval pass; `_Decimal32/64/128` (C23) needs a software
  decimal runtime and `_BitInt(N)` a width-masked lowering вЂ” both nice-to-have вќЊ now
  (see the Beyond-C99 table rows and the nice-to-have list below).
- **Header-version hygiene** вЂ” вњ… done for every case that applies:
  `<stdbool.h>`'s `bool`/`true`/`false` macro bodies are gated on
  `__STDC_VERSION__ < 202311L`, so under c23 they don't shadow the keywords
  (`#ifdef bool` is correctly false; only `__bool_true_false_are_defined`
  survives) вЂ” matching gcc. Fixture `stdbool-c23/`, unit tests
  `StdboolGatingTests`. `<assert.h>`'s `static_assert`, `<stdalign.h>`'s
  `alignas`/`alignof`, and `<stdnoreturn.h>`'s `noreturn` follow the same
  pattern (defined for C11 в‰¤ `__STDC_VERSION__` < C23; above that the C23
  keywords take over via rule-2 promotion). Gating `<stdatomic.h>` /
  `<stdint.h>` bodies on a *minimum* version would restrict an older dialect вЂ”
  against dotcc's permissive-by-default stance (feature-level rejection is
  `-pedantic`'s job via `DialectGate`, not the header's).

## Synthetic system headers + runtime

dotcc ships its own copies of the C95/C99/C11 standard headers (`stdio.h`, `stdlib.h`, `stddef.h`, `stdbool.h`, `stdint.h`, `inttypes.h`, `limits.h`, `float.h`, `assert.h`, `ctype.h`, `errno.h`, `iso646.h`, `setjmp.h`, `time.h`, `math.h`, `tgmath.h`, `string.h`, `threads.h`, `complex.h`, вЂ¦) plus the minimal POSIX set (`unistd.h`, `fcntl.h`, `poll.h`, `sys/types.h` / `sys/stat.h` / `sys/time.h` / `sys/select.h` / `sys/socket.h`, and the BSD-sockets headers `netinet/in.h` / `netinet/tcp.h` / `arpa/inet.h` вЂ” the `sys/`, `netinet/`, `arpa/` subdirectories embed under slash-qualified logical names) AND its libc implementations, both as **real files in source control**, both embedded into `DotCC.Lib.dll` so the compiler can serve them at emit time without any runtime disk I/O. Same model as clang's `lib/clang/<ver>/include/` tree, just loaded from the assembly manifest.

**Two parallel embeddings (see `DotCC.Lib.csproj`):**

| Source location | Logical name in manifest | Loader | Consumed by |
|---|---|---|---|
| `DotCC.Lib/include/*.h` | `DotCC.SystemHeaders.<filename>` | `Compiler.LoadEmbeddedSystemHeaders` | Preprocessor: resolves `#include <math.h>` etc. against the embedded files |
| `..\DotCC.Libc\*.cs` | `DotCC.Runtime.<filename>` | `Compiler.LoadRuntimeBlock` | `BuildShell`: splices the runtime source into the emitted program's type-decls section |

The runtime embedding is what makes dotcc's emit **single-source-of-truth**. The `.cs` files under `DotCC.Libc/` (`Libc.cs`, `StringLib.cs`, `StdlibLib.cs`, `StdioLib.cs`, `ErrnoLib.cs`, `TimeLib.cs`, `InttypesLib.cs`, `MathLib.cs`, `ComplexLib.cs`, `CTypeLib.cs`, `AssertLib.cs`, `SetJmpLib.cs`, `ThreadsLib.cs`, `PrintfBuilder.cs`, `ScanfReader.cs`, `SprintfBuilder.cs`, вЂ¦; picked up by a `*.cs` glob) compile into `DotCC.Libc.dll` for unit testing AND are loaded from the manifest at emit time. `LoadRuntimeBlock` strips file-scope artifacts (`#nullable enable`, `using` directives, `namespace DotCC.Libc;`) so the contained class declarations land cleanly at file scope in the emitted program. Every emitted file then has `using static Libc;` at the top, which brings `printf` / `malloc` / `sin` / `cos` / `sqrt` / etc. into scope by bare name.

**Header guards use the standard glibc/MSVC convention** (`_STDIO_H`, `_MATH_H`, вЂ¦) rather than a dotcc-specific prefix, so portable code that does `#ifdef _MATH_H` to detect whether the header has been pulled in works correctly.

**Why this design**: emitted file-based programs (`#:property AllowUnsafeBlocks=true`) stay self-contained вЂ” you can `dotnet run hello.cs` with no `<PackageReference>` and no external assemblies вЂ” but the implementations live in real refactorable `.cs` files, not as raw-string literals in the compiler source. The migration path to a NuGet-distributed `DotCC.Libc` is a one-line change in `BuildShell` (drop the `{{runtimeBlock}}` splice, add `#:package DotCC.Libc@<ver>` to the file-based header).

**Large entry stack**: `BuildShell` runs the C entry point (`main`) on a dedicated thread with a **64 MB stack** (`new System.Threading.Thread(thunk, 64 * 1024 * 1024)`, joined synchronously; the exit code rides back through a captured local). A native C program runs on a multi-megabyte stack (Linux defaults to ~8 MB) and deeply-recursive C relies on that headroom; .NET's default ~1 MB stack overflows far sooner and a managed `StackOverflowException` is **uncatchable** (it tears down the process). The classic casualty is a program with its own recursion guard calibrated to a native stack вЂ” e.g. Lua's VM caps nested C calls at `LUAI_MAXCCALLS` (200) and raises a *catchable* "C stack overflow" at the limit; on a 1 MB stack the emitted frames overflow at ~100 and crash before the guard trips. Reserving 64 MB (virtual address space, not committed memory) restores the native headroom so such programs hit their own limits and fault gracefully. Unit tests `EntryStackThreadTests`.

**MSVC oracle interop**: when `cl.exe` compiles the same `.c` source, it uses its own `<math.h>` (not ours). Both compilers declare the same C99 surface, so the differential test still produces matching output. Each side using its own copy of the standard headers is by design, not a bug.

### Design note вЂ” why dotcc does *not* impersonate clang to CMake

There are two levels of "pretend to be a clang compiler", and dotcc deliberately takes one and not the other:

- **CLI-convention level вЂ” adopt it (and we do).** Accept `-I`/`-D`/`-o`/`-c`/`-std=`, the glued spellings, and ignore-with-a-warning unmodeled flags. The more clang-shaped the CLI, the less any build system has to know about us вЂ” this is exactly why `-I`/`-D` threading needed *zero* CLI work.
- **Compiler-*identity* level (`CMAKE_C_COMPILER_ID "Clang"`) вЂ” don't.** Claiming to be Clang loads CMake's built-in Clang/platform modules, which assume a *native* toolchain: a real linker line (`clang -o exe a.o b.o`), injected `-O3 -DNDEBUG -fPIC`, and the `-MD -MF` depfile protocol the Ninja/Make generators rely on. We can honor none of those as-is вЂ” our objects are `.cs` fragments and our "link" is mergeв†’buildв†’launcher. So we declare our own compiler ID + `Generic` system + explicit rules instead: clang-shaped ergonomics, no native baggage.

Precedent: emscripten's `emcc` *is* clang underneath and transpiles Cв†’wasm (as we do Cв†’.NET), yet integrates with CMake as its **own** compiler ID via `Emscripten.cmake` вЂ” not by impersonating Clang вЂ” for exactly these reasons. dotcc is in the canonical spot for a transpiler. "Be more clang-like" pays off as concrete, bounded features (the `-MD` depfiles above), not as identity.

## Standards interop (autoconf / `./configure`)

A real C compiler has to be driveable by feature-detection tooling. The autoconf model is: `$CC -c conftest.c` runs against tiny probe programs, and the exit code drives `#define HAVE_X 1` macros in `config.h`. dotcc participates in this naturally because its CLI is clang-shaped (`-c`, `-E`, `-I`, `-D`, `-o`) and it exits non-zero on parse failure.

**Probes that need workarounds:**

- **Dialect flags** вЂ” `-std=c99`, `-pedantic`, and `-pedantic-errors` are recognized (`-pedantic`/`-pedantic-errors` drive the dialect-rejection gate). **Unknown flags (`-Wall`, `-O2`, `-g`, `-fPIC`, `-march=native`, вЂ¦) are now accepted-and-ignored** (warned once, partitioned out of the input-file list) rather than erroring вЂ” so a `./configure`/`make` driving dotcc survives the gcc/clang flag grab-bag. The one residual case: a space-separated unknown flag that consumes a following token (`-march native` rather than `-march=native`) would mis-read `native` as an input file; the `=`-joined forms are fine.
- **`-fsanitize=address`** вЂ” actually *modeled* (not ignored): routes the emitted program's `malloc`/`calloc`/`realloc`/`free` through a checked debug heap вЂ” a 32-byte `[magic|size]` header keeps the user pointer 32-aligned and a trailing redzone canary catches a write past the allocation, so `free` flags a bad/double free or an overflow with a managed stack trace at the offending call. A heap-only subset of clang's ASan (no stack/global redzones, no shadow memory). The shell calls `Libc.EnableDebugHeap()` once before `main`; `DOTCC_DEBUG_HEAP=1` is the no-recompile runtime override, and `DOTCC_DEBUG_HEAP_SCAN=1` adds a full live-block redzone sweep on every alloc/free (catches an overflow into a never-freed block, e.g. a guest GC arena). Other `-fsanitize=` kinds warn and are ignored. Off by default вЂ” zero behavioral change, one inert branch in malloc/free.
- **Compiler identity macros** вЂ” `__GNUC__` / `__clang__` / `_MSC_VER` aren't predefined. Codebases probing for these correctly land in the generic-compiler fallback, which is usually a safe default. We could predefine a `__DOTCC__` macro so codebases can target dotcc-specific paths if needed.
- **`AC_CHECK_LIB([m], [sin])`** вЂ” link-time probes. dotcc now **accepts `-l`/`-L`** (dynamic import mode) and static `.a`/`.lib` archives, though it still emits no native `.o`/`.a` objects. For this specific probe `sin` is runtime-provided (declared in the synthetic `<math.h>`, so it lexes in the line band and resolves to the embedded runtime via `using static Libc;`) вЂ” the `-lm` is accepted and the probe passes at compile time regardless of a real libm. A genuinely *external* symbol the probe declares itself (no system header) becomes an import candidate, bound GOT-style against the named `-l` library. See the `-lfoo` tooling row.
- **`AC_RUN_IFELSE`** вЂ” probes that compile AND execute the probe program. Works in principle (emitted `.cs` runs via `dotnet run`), but autoconf expects a native executable in `./conftest.exe`. A `dotcc --wrap` mode that emits a shim shell script invoking `dotnet run` on the emit would close this gap. Lower priority вЂ” most probes are compile-only.
- **Error-message-parsing probes** вЂ” a few legacy probes grep gcc-style diagnostics (`"error: implicit declaration"` etc.). dotcc emits its own format (`"dotcc: parse failed: вЂ¦"`). Most modern probes check only exit codes; the legacy ones need either a `--diag-format=gcc` shim or codebase-level patches.

**Realistic next milestone**: pick a small autotools-using codebase (`libcsv`, a small zlib subset, a stb_*-style single-file thing) and run `CC=dotcc ./configure && make`. The first failure tells you where to invest. Most likely landing-pad: unrecognized dialect/warning flags вЂ” a 5-minute fix in `System.CommandLine`.

## Test corpus and oracles

**dotcc-managed fixtures** live in `DotCC.FunctionalTests/Fixtures/<name>/`. Each is a small idiomatic C program with a hand-written `expected-stdout.txt`. They're the smoke layer вЂ” they exercise a specific feature or interaction (`loop-break`, `fizzbuzz`, `fibonacci`, etc.) and grow as the language does. Adding one is mechanical: drop a folder, the theory test picks it up.

**MSVC oracle**: `DotCC.FunctionalTests/MsvcOracle.cs` finds Microsoft's C compiler via `vswhere.exe` в†’ `vcvars64.bat` в†’ `cl.exe`. When MSVC is available, `MsvcOracleTests` recompiles every fixture with cl and asserts that dotcc's stdout matches MSVC's byte-for-byte. This is **differential testing** вЂ” it catches dotcc emitter / libc divergences from real-C semantics that a hand-written expected file might miss (printf precision quirks, integer overflow, evaluation order). Skipped automatically when MSVC isn't on the host (non-Windows or no VS install). Adding a new fixture means it gets validated against MSVC for free вЂ” no per-fixture wiring.

**External corpora we could pull from (not yet)**:

- **GCC testsuite** вЂ” has a [torture-tests](https://gcc.gnu.org/onlinedocs/gccint/Torture-Tests.html) section that's specifically small, self-contained, compiler-stress-testing. License: GPL вЂ” vendoring needs care, but referencing for inspiration is fine.
- **tcc-test / lcc-test** вЂ” small compilers ship their own focused test suites; useful as a reference for what "MVP C compiler tests" look like.
- **`nothings/single_file_libs`** (Sean Barrett's stb_*) вЂ” these are real-world idiomatic C programs that intentionally stay simple. Compiling one of them end-to-end would be a meaningful milestone.
- **csmith** вЂ” random C program generator; useful once dotcc handles most of C99, for fuzz-testing.
- **Rosetta Code C entries** вЂ” varied small idiomatic programs across hundreds of tasks. Low-overhead source of new fixtures.

The right strategy is probably to grow the fixture corpus organically as features land (one fixture per feature, oracled by MSVC), and reach for external suites once the grammar covers most of C99.

## How to add a feature

1. **Grammar change**: edit `DotCC.Lib/c.lalr.yaml`. Add symbols (append; don't reorder existing entries), productions (with new `action: foo` names), and lexer rules. The source generator emits `C.Foo` AST records on next build вЂ” your visitor will get a compile error pointing at the missing `Visit(C.Foo n)`.
2. **Visitor lowering**: add `Visit(C.Foo n) => вЂ¦` in `DotCC.Lib/CSharpEmitter.cs`. Aim to emit syntactically clean C# that preserves C's observable semantics; reuse Libc helpers when available (`Printf`, `Malloc`, etc.).
3. **Libc surface**: if the feature needs a new runtime function (e.g. `strtol`), add it to `DotCC.Libc/Libc.cs` (routing to the obvious BCL primitive), plus a unit test in `DotCC.Tests/LibcTests.cs`.
4. **Fixture**: drop a `DotCC.FunctionalTests/Fixtures/<feature>/main.c` + `expected-stdout.txt`. The theory test discovers it automatically вЂ” no code change. Pick a fixture name that's descriptive (`loop-break`, `bitwise-mask`, `struct-point`).
5. **Dialect gate** (if dialect-sensitive): if the feature is newer than `c90` (or was *removed* by a standard), add a `DialectGate` call so `-pedantic` rejects it under an older `-std=` вЂ” see `DotCC.Lib/DialectGate.cs`. Introduced-in features call `Gate(era, вЂ¦)` / `RequireMin` at the natural layer (visitor, `TypeFromSpec`, `CPreprocessor`). **The currently-moot features will need this when they become parseable: the *removed* ones вЂ” K&R definitions and trigraphs (both removed C23) вЂ” which need a removed-in / `RequireMax` direction that `DialectGate` does not have yet (add it when the first one lands). (VLAs are рџљ« out of scope, so no gate needed.)** Add unit tests in the dialect-gating region of `DotCC.Tests/CompilerTests.cs`.
6. **Update this file**: flip the row from вќЊ to вњ…, mention the fixture in **Notes**.

The grammar conventions (precedence ladder, dangling-else handling, lexer ordering) live in `CLAUDE.md` under **Grammar conventions** вЂ” read that before touching `c.lalr.yaml`.
