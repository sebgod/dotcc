# C language support in dotcc

Running tracker of what dotcc's grammar and libc cover today. Update this when a feature lands (add the fixture / test reference in **Notes**), when one moves from тЭМ to ЁЯЯб to тЬЕ, or when something is decided out of scope (ЁЯЪл with reason).

**Default dialect: `c17`.** Select otherwise with `-std=<dialect>` (`c90`/`c99`/`c11`/`c17`/`c18`/`c23`) тАФ see the CLI flag table in CLAUDE.md. `-std=` predefines `__STDC_VERSION__` (and friends) so headers can branch via `#if __STDC_VERSION__ >= тАжL`, and drives rule-2 keyword promotion (`inline` becomes a keyword тЙеc99; `bool`/`true`/`false`/`nullptr`/`noreturn`/`typeof` only тЙеc23 тАФ gated on the monotonic `CDialect.Version`, which is keyed by ISO year). **By itself `-std=` stays permissive** тАФ the parser is dialect-agnostic, so `//` comments, `_Bool`, designated initializers etc. are accepted regardless. **`-pedantic` / `-pedantic-errors` opt into the *rejection* gate** (gcc model): features newer than the selected `-std=` are diagnosed as warnings / errors. So the year tags below (`C99` / `C11` / `C23`) are descriptive for the default path, and become enforcement gates under `-pedantic`. Gated today: `_Bool` / `long long` / `ll`-suffix / designated init / `for`-init decl / `__func__` / variadic macros / mixed declarations-and-statements / compound literals / hex float literals / array designators / `_Complex` / flexible array member (C99), `_Static_assert` / `_Noreturn` / anonymous struct/union members (C11), `enum : T` / `_Float128` / `#warning` / empty initializer `{}` (C23). Not gated: `//` comments (universal extension); K&R definitions are moot (not parsed) and VLAs are out of scope. Always-modern *output* is unaffected by `-std=` тАФ dotcc emits real C# enums etc. regardless of input dialect.

## Coverage at a glance

Bird's-eye scorecard тАФ the detailed per-area tables below are the source of truth; this just rolls them up. **dotcc covers essentially all of C89/C99, and most of C11/C23 that maps cleanly onto .NET.** Output is always modern C# regardless of input dialect.

| Area | Standing | Covered | Gaps / out of scope |
|---|---|---|---|
| **Lexical** | тЬЕ Complete | every comment, identifier, integer (dec/hex/octal/binary, `u`/`l` suffixes, digit separators), float (incl. hex-float and the `L` suffix), char, and string (adjacent concatenation, high-byte) form | ЁЯЪл wide string/char literals, trigraphs, digraphs |
| **Types** | тЬЕ Near-complete | `int`тАж`long long`, `float`/`double`/`long double`, `_Complex`, `_Float128`, pointers, arrays (multi-dim, decay, ptr-to-array), `struct`/`union`/`enum`/`typedef`, function pointers, `const`/`volatile`/`restrict`, `sizeof(type)` + `sizeof expr`, bit-fields (value-faithful) | ЁЯЯб bit-fields (layout unpacked тАФ values correct), array compound literals (init position only) ┬╖ ЁЯЪл VLAs, `_BitInt(N)` |
| **Operators** | тЬЕ Near-complete | all arithmetic / relational / logical / bitwise / assignment / compound-assignment, `++`/`--`, ternary, comma, `sizeof`, cast, call, `.`/`->`, subscript | тЭМ `_Generic`, `_Alignof` |

| **Preprocessor** | тЬЕ Near-complete | `#include` (both forms), object + function-like macros (`##`, `#`, variadic), full `#if` constant-expr eval, `#ifdef`/`#else`/`#elif`/`#endif`, `#error`/`#warning`, `__FILE__`/`__LINE__`/`__func__`, multiple-include optimization | ЁЯЯб `#pragma` (only `once`) ┬╖ ЁЯЪл `#line` ┬╖ тЭМ `#elifdef`/`#elifndef`, `#embed` |

**C11 / C23 specifically** тАФ тЬЕ done: `_Bool`/`bool`, `true`/`false`/`nullptr`, `typeof`, `auto` inference, `_Atomic` + full `<stdatomic.h>`, `_Float128`, empty initializer `{}`, anonymous struct/union members, `_Noreturn`, binary + digit-separator literals; ЁЯЯб partial: `_Static_assert`/`static_assert` (parsed, inert тАФ not evaluated), `<threads.h>` subset. тЭМ roadmap: `_Generic`, `_Alignas`/`_Alignof`, `_Thread_local`, `constexpr`, `[[attributes]]`, `#embed`/`#elifdef`. ЁЯЪл out of scope: `_BitInt(N)`, wide literals, trigraphs/digraphs.

ЁЯЯб entries in the last column are **implemented but partial** тАФ they work for the common case and fail loudly outside it (see the per-area table for the caveat); they are *not* missing. E.g. `setjmp`/`longjmp` handles the standard `if (setjmp(env)) тАж` idiom (and multi-frame unwind), just not `switch (setjmp(тАж))` or value-capture.

**No known silent miscompiles** тАФ every gap fails loudly at parse/lex time, and `-pedantic` turns dialect violations into diagnostics.

## Lexical

| Feature | Status | Notes |
|---|---|---|
| Line comments `// тАж` | тЬЕ | `LINE_COMMENT` lexer rule, action `ignore` |
| Block comments `/* тАж */` | тЬЕ | `BLOCK_COMMENT` lexer rule, the canonical C-comment regex `/\*([^*]\|\*+[^*/])*\*+/` (alternation, supported since LALR.CC тЙе 4.0.0). A run of stars before the slash belongs to the terminator, so a comment closed with `**/` (Lua's doc-comment style, e.g. lfunc.c's `newupval`) ends there. (The earlier no-alternation form `(\*[^/][^*]*)*` mis-scanned `**/` тАФ the slash fell through into the body тАФ and ran the comment on to the next `*/`, swallowing the code between.) Fixture `block-comment-close/`, unit tests `BlockCommentTests`. |
| Identifiers `[a-zA-Z_][a-zA-Z_0-9]*` | тЬЕ | `ID` token. A C identifier that collides with a **C# reserved keyword** (`new`, `lock`, `is`, `string`, `this`, `ref`, `object`, `in`, `out`, тАж) is `@`-escaped on emit (`@new`) тАФ consistently at every declaration and reference (locals, params, function names + calls, struct fields + member access, labels, enum constants), since the escape is a pure function of the name. Fixtures `cs-keyword-idents/`, `keyword-true-false-ident/`. **Not escaped:** `null` (dotcc emits it as the bare C# `null` literal тАФ the only expression that implicitly converts to any pointer type тАФ so a variable named `null` is a residual edge; `true`/`false` are NO LONGER in this bucket тАФ they lower to `1`/`0` and so escape correctly), and **type names** (struct/enum/union tags, typedef-names) тАФ a `struct lock` tag isn't escaped yet. |
| Decimal int literal | тЬЕ | `NUM` token |
| Hex int literal `0xFF` | тЬЕ | Lexer rule above NUM (longest-match wins); visitor passes through тАФ C# accepts identical syntax. Fixture `bitwise/` |
| Octal int literal `0755` | тЬЕ | Lexed by the decimal `NUM` rule (a `0`-prefixed run), then **converted** in `Visit(C.Num)` to its value (`0755` тЖТ `493`) тАФ C# has no octal syntax (a leading `0` is plain decimal there), so emitting verbatim would silently mean 755. Digits validated `0`тАУ`7` (`0789` тЖТ clear error). C89, no dialect gate. Fixture `octal-binary-literals/`. |
| Binary int literal `0b1010` | тЬЕ | C23. Lexer rules `0[bB][01]+` (┬▒ suffix) above the decimal rule; C# accepts `0bтАж` verbatim so the visitor passes it through. Gated as C23 under `-pedantic`. Fixture `octal-binary-literals/`. |
| Digit separators `1'000'000` | тЬЕ | C23. Each NUM digit-run rule allows a single `'` between digits (`D('?D)*` тАФ never leading/trailing/doubled), in decimal, hex, and binary. The visitor strips them (C# uses `_` and needs none) before evaluating. Gated as C23 under `-pedantic`. Fixture `digit-separators/`. |
| Int suffixes `123u`, `123L`, `123ull` | тЬЕ | Lexer accepts any combination of `u`/`U`/`l`/`L` (one or more) after a decimal/hex/binary/octal digit run. Visitor's `CsIntSuffix` collapses to C#'s form: `u` тЖТ `u` (compiler resolves uint/ulong), one-or-more `L` тЖТ `L` (long; C# has no `ll`), `u` + `L`s тЖТ `UL` (ulong). |
| Float literal `1.5`, `1.5e10`, `1.5f` | тЬЕ | `FLOAT` token (mandatory `.` to disambiguate from `NUM`) |
| Float literal suffix `1.5L` (long double) | тЬЕ | The suffixed FLOAT lexer rule accepts `[fFlL]`. `f`/`F` passes through to C# verbatim (float); `l`/`L` is C's long-double suffix тЖТ `Visit(C.Flt)` strips it and emits a bare `double` literal (mirroring dotcc's `long double` тЖТ C# `double` mapping тАФ C# has no float `L` suffix). Works after an exponent too (`6.0e2L`). Fixture `float-literal-suffix/` (gcc-oracle-validated). |
| Hex float literal `0x1.8p3` (C99) | тЬЕ | `0xH.HpE` тАФ value = (hex mantissa) ├Ч 2^(decimal exponent); the `p`/`P` exponent is mandatory (so longest-match cleanly separates it from a hex int). Three FLOAT lexer rules cover the int-part/fraction-part variations. C# has **no** hex-float literal, so `Visit(C.Flt)` parses the value (`ParseHexFloat`) and emits a decimal `double` (or `float` for the `f` suffix; `l`/`L` тЖТ double). Exactly-representable constants round-trip bit-for-bit; extreme exponents could differ by an ULP via `Math.Pow`. Gated C99 under `-pedantic`. Fixture `hex-float-literals/` (gcc-oracle-validated). |
| Char literal `'a'`, `'\n'`, `'\033'`, `'\x1b'` | тЬЕ | Five `CHAR` lexer rules (longest-match: hex `'\xNN'`, 3- and 2-digit octal, single-escape incl. 1-digit octal, plain). The visitor **decodes** the escape to its byte value: a plain printable char stays `(byte)'c'`, everything else (named escape, octal, hex, control) тЖТ `(byte)N`. Full escape set: `\a \b \e \f \n \r \t \v \0 \NNN \xNN \\ \' \" \?` (`\e` is the GNU ESC). Char bytes 0тАУ255 all fine (it's a `(byte)` cast). Fixtures `small-ops/`, `string-char-escapes/`. |
| String literal `"тАж"` + adjacent concatenation `"a" "b"` | тЬЕ | `STRING` token regex `"(\\["\\nrtbf0v'/aex0-9]\|[^"\\])*"` (open quote, then backslash-escapes or non-quote-non-backslash, then close). **Adjacent literals concatenate** (C phase 6) via a left-recursive `StringSeq` тЖТ one `Primary`. Each segment's escapes are **decoded independently** to bytes, then concatenated, then re-emitted as a single greedy-safe C# `u8` literal: named escapes (`\n \t \r \\ \"`), and everything else as `\xHH` тАФ and a `\xHH` is never left adjacent to a literal hex digit (C#'s `\x` is greedy), so octal/hex escapes and `"\x1b" "abc"`-style boundaries are correct. Non-ASCII source chars pass through (the u8 literal UTF-8-encodes them, matching C's UTF-8 source bytes). Fixtures `string-escape-quotes/`, `string-char-escapes/`. **High bytes:** a decoded escape byte > 0x7F (`"\xff"`, `"\377"`) can't be a single byte in a u8 literal (C# UTF-8-encodes it to two), so such strings lower to a constant byte-array `L(new byte[]{ 0xFF, тАж, 0 })` carrying the exact C bytes тАФ Roslyn RVA-optimizes a constant `byte[]` in `ReadOnlySpan<byte>` position, so `L()` still pins a fixed-address pointer (no alloc, no GC move), identical to the u8 case. All-ASCII strings keep the readable u8 literal. Fixture `string-high-byte-escapes/` (gcc-oracle-validated). Relies on IRx alternation in LALR.CC. |
| Wide string literal `L"тАж"`, `u"тАж"`, `U"тАж"` | ЁЯЪл | dotcc is UTF-8-native тАФ wide types add no value |
| Escape sequences `\n \t \\ \" \xNN` | тЬЕ | Recognized inside string literals (above) and char literals; passed verbatim into the C# UTF-8 literal so C# decodes them. |
| Trigraphs `??=` etc. | ЁЯЪл | Removed in C23; never useful |
| Digraphs `<% %> :> :>` etc. | ЁЯЪл | Same reasoning |

|---|---|---|

| `float` | тЬЕ | C# `float` |
| `double` | тЬЕ | C# `double` |
| `long double` | тЬЕ | Lowered to C# `double` тАФ the CLI (dotcc's target) offers no wider IEEE float, so there's no native `long double` width to match; exactly mirrors the `long long` тЖТ `long` collapse. | `float`/`double`/`long double _Complex` (C99) | тЬЕ | Complex-number types, lowered to **`System.Numerics.Complex`** (double-backed). `_Complex` is a `_Capital_` keyword combined with a floating base in `ResolveTypeSpec` (a non-floating base like `int _Complex` is rejected); the lowercase `complex` is a `<complex.h>` macro. All three widths map to the one `Complex` type, so `float`/`long double _Complex` widen to double precision (documented, same shape as `long double`тЖТ`double`). **Arithmetic is free**: `Complex` supplies `+ тИТ * /` and an implicit conversion from `double`, so `z * z`, `z + 1.0`, `2.0 * I` compile with nothing special emitted. The imaginary unit `I` / `_Complex_I` is a header macro for `Libc.__dotcc_complex_I` (`Complex.ImaginaryOne`). The `<complex.h>` function surface (`creal`/`cimag`/`cabs`/`carg`/`conj`/`cproj`/`csqrt`/`cpow`/`cexp`/`clog`/`csin`тАж`ctanh`) routes to `Complex`'s static methods (`ComplexLib.cs`). Fixture `complex-numbers/` (gcc-oracle-validated; MSVC opts out тАФ its C frontend has no `_Complex`). **Gaps:** the explicit `тАжf`/`тАжl` function variants aren't declared yet (they'd alias the double forms); `_Imaginary` is unsupported. |
| `_Float128` / `__float128` (C23) | тЬЕ | IEEE-754 **binary128**, lowered to the MIT software `DotCC.Libc.Float128` (clean-room, every op validated bit-for-bit against gcc's binary128). Full correctly-rounded arithmetic `+ тИТ * /`, `sqrt`, `fma`; relational `< > <= >=` (NaN-unordered); int/`double` conversions; and `printf` `%Lf`/`%Le`/`%Lg` at full quad precision (own BigInteger decimal formatter, no narrowing). **Complete `<math.h>` surface** тАФ algebraic (`floor`/`ceil`/`trunc`/`round`/`copysign`/`fmod`/`remainder`/`cbrt`/`hypot`/`scalbn`/`ilogb`/`nextafter`) and transcendental (`exp`/`log`/`exp2`/`exp10`/`expm1`/`log2`/`log10`/`log1p`/`pow`/`sin`/`cos`/`tan`/`asin`/`acos`/`atan`/`atan2`/`sinh`/`cosh`/`tanh`/`asinh`/`acosh`/`atanh`/`rootn`) тАФ implemented via a high-precision BigInteger fixed-point core and **validated against gcc's binary128 (`тАжl`) functions**: bit-exact for the correctly-rounded ops, тЙд2тАУ4 ULP for the transcendentals (gcc's own aren't correctly rounded). Also implements **`System.Numerics.IBinaryFloatingPointIeee754<Float128>`** (decimal Parse/ToString, conversions, generic-math interop). `__float128` is the GNU spelling (accepted by dotcc; gcc only defines it on x86, so the aarch64 gcc oracle covers `_Float128`). Fixtures `float128-basic/` (arithmetic) + `float128-print/` (`%Lf` of 1/3 to 40 places). MSVC's cl.exe has no `_Float128`, so those fixtures opt out of the MSVC oracle. |
| `void` | тЬЕ | C# `void`; only valid as return type or `void*` |
| `_Bool` / `bool` (C99 stdbool.h) | тЬЕ | `_Bool` is a TypeSpec keyword lowered to the integer-typed value struct **`Libc.CBool`** (NOT C# `bool`), so C's defining `_Bool` semantics hold at every boundary via implicit conversions: storing any scalar normalizes to 0/1, and a bool converts freely to `int` in arithmetic, assignment, return, argument, and `%d` positions (`int x = b;`, `b + 1`, `return b;` from an `int` function all compile and behave as C). `sizeof(_Bool)` is 1. Synthetic `stdbool.h` defines `bool` тЖТ `_Bool` and `true`/`false` тЖТ the integer constants `1`/`0` (their C values, normalized through `CBool`). Pointer stores work too: `_Bool b = somePointer;` (meaning `somePointer != NULL`) is covered by a `void*` тЖТ `CBool` conversion operator тАФ a typed `T*` reaches it through the standard `T* тЖТ void*` conversion (one standard + one user-defined step, which C# permits; C# does **not** forbid pointer conversion operators). Combinations with sign/size specifiers (`unsigned _Bool` etc.) throw `CompileException`. Fixtures `bool-stdbool/`, `bool-as-int/`, `bool-from-pointer/`. Under **`-std=c23`** `bool` is a first-class keyword: `DialectKeywordRewriter` promotes bare `bool` тЖТ `_Bool` with no `<stdbool.h>` include needed (pre-C23 it stays an ordinary identifier, so the macro path and old code using `bool` as a name both keep working). Including `<stdbool.h>` under c23 is harmless тАФ the synthetic header's `bool`/`true`/`false` macro bodies are gated on `__STDC_VERSION__ < 202311L` (vestigial in C23), so they don't shadow the keywords and `#ifdef bool` is correctly false; only `__bool_true_false_are_defined` survives. Matches gcc; fixture `stdbool-c23/`. |
| `true` / `false` / `nullptr` (C23 keywords) | тЬЕ | Under **`-std=c23`** these are first-class keyword constants тАФ `DialectKeywordRewriter` promotes them onto dedicated `TRUE`/`FALSE`/`NULLPTR` grammar terminals (no lexer rule тАФ same rule-2 gating as `bool`), lowering to the integer literals `1`/`0` (normalized through `CBool` when stored to a `_Bool`) and C# `null`, with no `<stdbool.h>`/`<stddef.h>` include. Pre-C23 they stay ordinary identifiers; since `true`/`false` no longer emit as C# keywords, a variable named `true`/`false` is correctly `@`-escaped (`int true = 5;` тЖТ `int @true = 5;` тАФ fixture `keyword-true-false-ident/`). Fixture `c23-constants/` (gcc oracle under `-std=c2x`; opts out of MSVC, which has no C23 frontend). |
| Pointer types `T*`, `T**`, тАж | тЬЕ | Composes left-to-right via `TypePtr` production |
| Array decl `T arr[N]` | тЬЕ | Lowered to `T* arr = stackalloc T[N]` so block-scoped automatic arrays match C's lifetime + subscript semantics; fixture `array-sum/`. A non-constant 1-D extent (`int a[n]`) lowers to `stackalloc T[n]` (VLA-ish). |
| Multi-dimensional array `T a[D1][D2]тАж` | тЬЕ | C# `stackalloc` is 1-D, so dotcc **flattens** to one contiguous block `stackalloc T[D1*D2*тАж]` (row-major, like C) via the `ArrDims` dimension list, tracking the shape as a nested `CType.Arr`. A **partial** subscript `a[i]` (dimensions remain) rewrites to `a + i*stride` (stride = element count of the remaining sub-array); a **full** subscript `a[i][j]` is the flat element. The nested `sizeof` length idiom works (`sizeof(a)/sizeof(a[0])` = rows, `sizeof(a[0])/sizeof(a[0][0])` = cols). All dimensions must be compile-time constants (a non-constant multi-dim extent errors clearly). **Nested-brace initializers** (`int m[2][2] = {{1,2},{3,4}}`) work тАФ see the initializer row below. Fixture `multidim-array/`. |
| Array param decay `T arr[]` / `T arr[N]` | тЬЕ | Both forms lower to `T* arr` per C99 ┬з6.7.5.3p7. The size in the sized form is informational only (C compilers don't enforce it at call sites). Fixture `array-param-decay/` |
| File-scope (global) array `T a[N]` | тЬЕ | A file-scope array has static storage duration and decays to a pointer, so dotcc lowers it to a `T*` field in `DotCcGlobals` backed by a **pinned, rooted managed array** тАФ `Libc.GlobalArrayZeroed<T>(n)` (no initializer; C zero-inits static storage) or `Libc.GlobalArrayFrom<T>(new T[]{тАж})` (brace / string init). An **array OF POINTERS** (`const char *const names[] = {тАж}`) can be neither a C# generic type argument nor an array element (CS0306 / CS0611), so it's stored as a pinned `nint[]` (pointer-width) тАФ `(T**)Libc.GlobalArrayFrom<nint>(new nint[]{ (nint)тАж, тАж })` тАФ reinterpreting the base as `T**`. **`extern T a[];` / `extern T a[N];`** declare without defining тАФ no field emitted (storage lives in another TU); the name's type is registered so same-TU references resolve. Fixture `file-scope-arrays/`. (This lifted the old "static arrays / aggregate-init statics not yet" limitation in the storage-class row.) |

## libc (`DotCC.Libc/`)

The runtime surface dotcc-emitted programs link against. Each function routes to a BCL primitive. Implementations live in `DotCC.Libc/*.cs`; the same source compiles into `DotCC.Libc.dll` for unit tests AND is embedded as a resource into `DotCC.Lib.dll` (spliced into every emitted program by `BuildShell` тАФ single source of truth). Once `DotCC.Libc` is published to NuGet, the embedding goes away and emitted programs reference the package directly.

**Design rule: reentrant by default.** Where the C standard offers a stateful and a reentrant variant of the same function (`strtok` / `strtok_r`, `rand` / `rand_r`, `asctime` / `asctime_r`, `localtime` / `localtime_r`, etc.), dotcc implements the reentrant form as the primitive and exposes the stateful form (if at all) as a thin wrapper using `[ThreadStatic]` storage. Hidden static state in libc is a notorious source of multithreading bugs; emitted dotcc programs target .NET where threading is cheap and idiomatic, so the reentrant shape is the right default. Real C programs that need the C89 stateful form keep working through the wrapper, but the docs steer users to the `_r` variant.

### `stdio.h`

**FILE model.** A C `FILE` is a tiny opaque struct (`Libc.FILE`, one slot index) and `FILE*` stays a **genuine pointer** тАФ so `NULL`, `==`, and `if (fp)` work through the normal pointer machinery with no emitter special-casing (`FILE` is seeded in `Compiler.PredefinedTypeNames`, like `div_t`). The slot indexes a managed table (`FileSlot`): slots 0/1/2 are `stdin`/`stdout`/`stderr` and route text through `Console.In`/`Out`/`Error` **resolved per call** (so redirection / the functional harness's capture are honored тАФ never the raw OS handle); `fopen`'d slots wrap a `FileStream` (binary, seekable). `freopen` swaps a slot's backing while keeping the same `FILE*` identity тАФ the standard way (C99 7.19.5.4) to point `stdin`/`stdout` at a file. Implementations in `FileLib.cs`; unit tests `LibcFileTests.cs`; fixture `file-io-roundtrip/`.

## Dialect gating (rewriter / grammar / visitor)

How dotcc decides whether a feature is available under a given `-std=` тАФ the
model folded in from the former `HANDOVER-std-option.md`. `-std=` predefines
`__STDC_VERSION__` and drives keyword promotion; by itself it stays
**permissive** (the parser is a superset of every dialect), and `-pedantic` /
`-pedantic-errors` turn the per-feature year tags into warnings / errors. Three
layers carry the gating; the right one depends on the feature's *shape*:

1. **Rewriter** (`DialectKeywordRewriter`) тАФ a **new keyword spelled like an
   identifier** (`bool`, `true`, `false`, `nullptr`, `static_assert`, `noreturn`,
   `typeof`). These are *legal identifiers* in older dialects (`int true = 5;` is
   valid C99), so the grammar can't always treat them as keywords тАФ it would
   parse-error on valid old code before the visitor runs. The `ID тЖТ keyword`
   promotion itself carries the `MinVersion`: promoted at/above the introducing
   standard, left an identifier below it (which also lets the pre-C23
   `<stdbool.h>` / `<stddef.h>` macro supply the meaning). One `MinVersion` covers
   both directions at once. (`inline` is the same shape, gated тЙеc99.)
2. **Grammar** (`c.lalr.yaml`) тАФ **genuinely new syntax** that can't collide with
   an identifier (`_BitInt(N)`, `typeof`, `_Generic`, `[[attributes]]`). Parsed
   always (the superset); *availability* gated in the visitor.
3. **Visitor** (`CSharpEmitter`, via `DialectGate`) тАФ *restrictions*: a construct
   the grammar accepts but an older `-std=` shouldn't (designated init / `long
   long` under c90, тАж) becomes a `-pedantic` diagnostic. Also semantic *flips*
   like C23 `auto` (storage class тЖТ type inference), branching on the threaded
   `CDialect`.

`_Capital_` keywords (`_Bool`, `_Static_assert`, `_Noreturn`, `_Atomic`,
`_Alignas`, тАж) are reserved to the implementation in *every* dialect, so they
**always lex as keywords**; their *availability* can still be visitor-gated. We
deliberately keep dialect awareness OUT of LALR.CC itself тАФ a single client
doesn't justify the table-doubling; revisit only with a second
standards-versioned client.

**Residual keyword TODOs** (status mirrors the Beyond-C99 / dialect-gaps tables):
- `_Thread_local` / `thread_local`, `_Alignas` / `alignas`, `_Alignof` / `alignof`
  (C11) тАФ **not built; weigh value first.** No faithful lowering in the current
  emit shape: `[ThreadStatic]` is field-only (no per-local TLS), and there's no
  per-local alignment hook on a `stackalloc`. They'd only stub.
- `_Generic` (C11) тАФ the common case (`tgmath.h`) is already covered by C#
  overload resolution; low priority.
- `constexpr` (C23) needs a const-eval pass; `_Decimal32/64/128` (C23) needs a
  decimal runtime. `_BitInt(N)` is ЁЯЪл (no faithful .NET width).
- **Header-version hygiene** тАФ тЬЕ done for the one case that applies:
  `<stdbool.h>`'s `bool`/`true`/`false` macro bodies are gated on
  `__STDC_VERSION__ < 202311L`, so under c23 they don't shadow the keywords
  (`#ifdef bool` is correctly false; only `__bool_true_false_are_defined`
  survives) тАФ matching gcc. Fixture `stdbool-c23/`, unit tests
  `StdboolGatingTests`. The other headers the original handover listed don't
  apply: `<stdalign.h>` / `<stdnoreturn.h>` aren't shipped, and gating
  `<stdatomic.h>` / `<stdint.h>` bodies on a *minimum* version would restrict an
  older dialect тАФ against dotcc's permissive-by-default stance (feature-level
  rejection is `-pedantic`'s job via `DialectGate`, not the header's).

## Out of scope (won't implement)

Listed here so we don't relitigate them. All marked ЁЯЪл above.

- **Variable-length arrays (VLAs)** тАФ rarely used, made optional in C11, and a managed-runtime mismatch (unbounded stack growth). A 1-D runtime extent incidentally lowers to `stackalloc`, but full VLA support isn't pursued.
- **Trigraphs / digraphs** тАФ removed in C23, never useful.
- **Wide string/char literals** (`L"тАж"`, `wchar_t`, `wprintf`) тАФ dotcc is UTF-8-native; wide types add no value and complicate the lowering.
- **`signal.h`** тАФ POSIX signal model doesn't map onto .NET.
- **`gets`** тАФ removed in C11; security disaster.
- **Annex K bounds-checked interfaces** тАФ almost no real-world C uses them.
- **`_BitInt(N)`** тАФ only well-defined for two widths on .NET; not worth the complexity.

## Synthetic system headers + runtime

dotcc ships its own copies of the C95/C99/C11 standard headers (`stdio.h`, `stdlib.h`, `stddef.h`, `stdbool.h`, `stdint.h`, `inttypes.h`, `limits.h`, `float.h`, `assert.h`, `ctype.h`, `errno.h`, `iso646.h`, `setjmp.h`, `time.h`, `math.h`, `tgmath.h`, `string.h`, `threads.h`, `complex.h`) AND its libc implementations, both as **real files in source control**, both embedded into `DotCC.Lib.dll` so the compiler can serve them at emit time without any runtime disk I/O. Same model as clang's `lib/clang/<ver>/include/` tree, just loaded from the assembly manifest.

**Two parallel embeddings (see `DotCC.Lib.csproj`):**

| Source location | Logical name in manifest | Loader | Consumed by |
|---|---|---|---|
| `DotCC.Lib/include/*.h` | `DotCC.SystemHeaders.<filename>` | `Compiler.LoadEmbeddedSystemHeaders` | Preprocessor: resolves `#include <math.h>` etc. against the embedded files |
| `..\DotCC.Libc\*.cs` | `DotCC.Runtime.<filename>` | `Compiler.LoadRuntimeBlock` | `BuildShell`: splices the runtime source into the emitted program's type-decls section |

The runtime embedding is what makes dotcc's emit **single-source-of-truth**. The `.cs` files under `DotCC.Libc/` (`Libc.cs`, `StringLib.cs`, `StdlibLib.cs`, `StdioLib.cs`, `ErrnoLib.cs`, `TimeLib.cs`, `InttypesLib.cs`, `MathLib.cs`, `ComplexLib.cs`, `CTypeLib.cs`, `AssertLib.cs`, `SetJmpLib.cs`, `ThreadsLib.cs`, `PrintfBuilder.cs`, `ScanfReader.cs`, `SprintfBuilder.cs`, тАж; picked up by a `*.cs` glob) compile into `DotCC.Libc.dll` for unit testing AND are loaded from the manifest at emit time. `LoadRuntimeBlock` strips file-scope artifacts (`#nullable enable`, `using` directives, `namespace DotCC.Libc;`) so the contained class declarations land cleanly at file scope in the emitted program. Every emitted file then has `using static Libc;` at the top, which brings `printf` / `malloc` / `sin` / `cos` / `sqrt` / etc. into scope by bare name.

**Header guards use the standard glibc/MSVC convention** (`_STDIO_H`, `_MATH_H`, тАж) rather than a dotcc-specific prefix, so portable code that does `#ifdef _MATH_H` to detect whether the header has been pulled in works correctly.

**Why this design**: emitted file-based programs (`#:property AllowUnsafeBlocks=true`) stay self-contained тАФ you can `dotnet run hello.cs` with no `<PackageReference>` and no external assemblies тАФ but the implementations live in real refactorable `.cs` files, not as raw-string literals in the compiler source. The migration path to a NuGet-distributed `DotCC.Libc` is a one-line change in `BuildShell` (drop the `{{runtimeBlock}}` splice, add `#:package DotCC.Libc@<ver>` to the file-based header).

**MSVC oracle interop**: when `cl.exe` compiles the same `.c` source, it uses its own `<math.h>` (not ours). Both compilers declare the same C99 surface, so the differential test still produces matching output. Each side using its own copy of the standard headers is by design, not a bug.

## Standards interop (autoconf / `./configure`)

A real C compiler has to be driveable by feature-detection tooling. The autoconf model is: `$CC -c conftest.c` runs against tiny probe programs, and the exit code drives `#define HAVE_X 1` macros in `config.h`. dotcc participates in this naturally because its CLI is clang-shaped (`-c`, `-E`, `-I`, `-D`, `-o`) and it exits non-zero on parse failure.

| `AC_CHECK_FUNCS([printf])` / `[malloc]` / `[free]` / `[strlen]` / `[strcmp]` / `[strcpy]` / `[memset]` / `[memcpy]` / `[sin]` / `[cos]` / `[sqrt]` / `[pow]` / `[atoi]` / `[strtol]` / `[strchr]` / `[strstr]` / `[strtok]` / `[abs]` / `[qsort]` / `[bsearch]` / `[getenv]` / `[system]` / `[putchar]` / `[strerror]` / `[time]` / `[clock]` / `[strtoimax]` / `[fopen]` / `[fread]` / `[fwrite]` / `[fseek]` / `[fclose]` / `[remove]` / `[rename]` / `[tmpfile]` / тАж | тЬЕ HAVE_X=1 | Declared in our synthetic headers; the probe links because `using static Libc;` makes them resolvable in the emitted C#. |

| `AC_CHECK_FUNCS([localtime])` / `[gmtime])` / `[mktime])` / `[strftime])` / `[asctime])` / `[ctime])` | тЬЕ HAVE_X=1 | The `struct tm` calendar family тАФ declared in `<time.h>`, resolvable via `using static Libc;`. |

**Probes that need workarounds:**

- **Dialect flags** тАФ `-std=c99`, `-pedantic`, and `-pedantic-errors` are recognized (`-pedantic`/`-pedantic-errors` drive the dialect-rejection gate). **Unknown flags (`-Wall`, `-O2`, `-g`, `-fPIC`, `-march=native`, тАж) are now accepted-and-ignored** (warned once, partitioned out of the input-file list) rather than erroring тАФ so a `./configure`/`make` driving dotcc survives the gcc/clang flag grab-bag. The one residual case: a space-separated unknown flag that consumes a following token (`-march native` rather than `-march=native`) would mis-read `native` as an input file; the `=`-joined forms are fine.
- **Compiler identity macros** тАФ `__GNUC__` / `__clang__` / `_MSC_VER` aren't predefined. Codebases probing for these correctly land in the generic-compiler fallback, which is usually a safe default. We could predefine a `__DOTCC__` macro so codebases can target dotcc-specific paths if needed.
- **`AC_CHECK_LIB([m], [sin])`** тАФ link-time probes. dotcc doesn't produce `.o`/`.a` and doesn't accept `-l`. Workaround: declare a synthetic libm and have the probe pass at compile-time even without real link semantics. The `sin` function is already in our embedded runtime so a no-op `-l` accept-and-ignore would work.
- **`AC_RUN_IFELSE`** тАФ probes that compile AND execute the probe program. Works in principle (emitted `.cs` runs via `dotnet run`), but autoconf expects a native executable in `./conftest.exe`. A `dotcc --wrap` mode that emits a shim shell script invoking `dotnet run` on the emit would close this gap. Lower priority тАФ most probes are compile-only.
- **Error-message-parsing probes** тАФ a few legacy probes grep gcc-style diagnostics (`"error: implicit declaration"` etc.). dotcc emits its own format (`"dotcc: parse failed: тАж"`). Most modern probes check only exit codes; the legacy ones need either a `--diag-format=gcc` shim or codebase-level patches.

**Realistic next milestone**: pick a small autotools-using codebase (`libcsv`, a small zlib subset, a stb_*-style single-file thing) and run `CC=dotcc ./configure && make`. The first failure tells you where to invest. Most likely landing-pad: unrecognized dialect/warning flags тАФ a 5-minute fix in `System.CommandLine`.

## Test corpus and oracles

**dotcc-managed fixtures** live in `DotCC.FunctionalTests/Fixtures/<name>/`. Each is a small idiomatic C program with a hand-written `expected-stdout.txt`. They're the smoke layer тАФ they exercise a specific feature or interaction (`loop-break`, `fizzbuzz`, `fibonacci`, etc.) and grow as the language does. Adding one is mechanical: drop a folder, the theory test picks it up.

**MSVC oracle**: `DotCC.FunctionalTests/MsvcOracle.cs` finds Microsoft's C compiler via `vswhere.exe` тЖТ `vcvars64.bat` тЖТ `cl.exe`. When MSVC is available, `MsvcOracleTests` recompiles every fixture with cl and asserts that dotcc's stdout matches MSVC's byte-for-byte. This is **differential testing** тАФ it catches dotcc emitter / libc divergences from real-C semantics that a hand-written expected file might miss (printf precision quirks, integer overflow, evaluation order). Skipped automatically when MSVC isn't on the host (non-Windows or no VS install). Adding a new fixture means it gets validated against MSVC for free тАФ no per-fixture wiring.

**External corpora we could pull from (not yet)**:

- **GCC testsuite** тАФ has a [torture-tests](https://gcc.gnu.org/onlinedocs/gccint/Torture-Tests.html) section that's specifically small, self-contained, compiler-stress-testing. License: GPL тАФ vendoring needs care, but referencing for inspiration is fine.
- **tcc-test / lcc-test** тАФ small compilers ship their own focused test suites; useful as a reference for what "MVP C compiler tests" look like.
- **`nothings/single_file_libs`** (Sean Barrett's stb_*) тАФ these are real-world idiomatic C programs that intentionally stay simple. Compiling one of them end-to-end would be a meaningful milestone.
- **csmith** тАФ random C program generator; useful once dotcc handles most of C99, for fuzz-testing.
- **Rosetta Code C entries** тАФ varied small idiomatic programs across hundreds of tasks. Low-overhead source of new fixtures.

The right strategy is probably to grow the fixture corpus organically as features land (one fixture per feature, oracled by MSVC), and reach for external suites once the grammar covers most of C99.

## How to add a feature

1. **Grammar change**: edit `DotCC.Lib/c.lalr.yaml`. Add symbols (append; don't reorder existing entries), productions (with new `action: foo` names), and lexer rules. The source generator emits `C.Foo` AST records on next build тАФ your visitor will get a compile error pointing at the missing `Visit(C.Foo n)`.
2. **Visitor lowering**: add `Visit(C.Foo n) => тАж` in `DotCC.Lib/CSharpEmitter.cs`. Aim to emit syntactically clean C# that preserves C's observable semantics; reuse Libc helpers when available (`Printf`, `Malloc`, etc.).
3. **Libc surface**: if the feature needs a new runtime function (e.g. `strtol`), add it to `DotCC.Libc/Libc.cs` (routing to the obvious BCL primitive), plus a unit test in `DotCC.Tests/LibcTests.cs`.
4. **Fixture**: drop a `DotCC.FunctionalTests/Fixtures/<feature>/main.c` + `expected-stdout.txt`. The theory test discovers it automatically тАФ no code change. Pick a fixture name that's descriptive (`loop-break`, `bitwise-mask`, `struct-point`).
5. **Dialect gate** (if dialect-sensitive): if the feature is newer than `c90` (or was *removed* by a standard), add a `DialectGate` call so `-pedantic` rejects it under an older `-std=` тАФ see `DotCC.Lib/DialectGate.cs`. Introduced-in features call `Gate(era, тАж)` / `RequireMin` at the natural layer (visitor, `TypeFromSpec`, `CPreprocessor`). **The currently-moot features will need this when they become parseable: the *removed* ones тАФ K&R definitions and trigraphs (both removed C23) тАФ which need a removed-in / `RequireMax` direction that `DialectGate` does not have yet (add it when the first one lands). (VLAs are ЁЯЪл out of scope, so no gate needed.)** Add unit tests in the dialect-gating region of `DotCC.Tests/CompilerTests.cs`.
6. **Update this file**: flip the row from тЭМ to тЬЕ, mention the fixture in **Notes**.

The grammar conventions (precedence ladder, dangling-else handling, lexer ordering) live in `CLAUDE.md` under **Grammar conventions** тАФ read that before touching `c.lalr.yaml`.
